# オープンクエスチョン

## ~~OQ-001: JSONL プロトコルの正確な仕様確認~~ → **解決: D-16**

**決定:** テスト駆動で検証する。仕様書ベースの事前確認ではなく、実 CLI との統合テストで不整合を検出・修正していく方針。設計段階でテスト戦略に「プロトコル互換性テスト」を組み込む。

---

## ~~OQ-002: hooks 機能のスコープ~~ → **解決: D-11**

**決定:** hooks 機能は後回し。コア機能を先に安定させ、将来フェーズで対応する。

---

## ~~OQ-003: V2 API の安定性~~ → **解決: D-10**

**決定:** V2（セッション維持）を Primary API として実装する。

**理由:** プロセス再利用によるコールドスタート回避が実用上不可欠。V1 の query はセッション上のワンショット操作として表現できる。SDK 側の V2 安定化には追随していく。

---

## ~~OQ-004: 構造化出力（outputFormat）の JSON Schema 表現~~ → **解決: D-12**

**決定:** 辞書型（`[String: Any]` 相当）で表現する。外部ライブラリは使用しない。外部依存ゼロの方針を堅持。

---

## ~~OQ-005: プロセスプール / プリウォーミング~~ → **解決: D-15**

**決定:** スコープ外。コア機能に集中し、必要になった時点で別パッケージとして検討する。

---

## ~~OQ-006: Linux での Foundation.Process の動作検証~~ → **解決: D-13**

**決定:** Linux サポートは後回し。初期ターゲットは macOS のみ。macOS で安定させてから Linux 対応を検討する。

---

## ~~OQ-007: パッケージ・モジュール命名とモジュール分割戦略~~ → **解決: D-14**

**決定:** 3モジュール分割（戦略 C）を採用する。

| モジュール | 役割 | 依存 |
|-----------|------|------|
| `AgentSDK` | protocol 定義・共通型 | なし |
| `AgentSDKClaudeCode` | Claude Code CLI 具象実装 | `AgentSDK` |
| `AgentSDKTesting` | MockTransport 等テスト支援 | `AgentSDK` |

---

## ~~OQ-008: protocol の existential vs generics~~ → **解決: D-8**

**決定:** generics（`some AgentClient` / `<T: AgentClient>`）を採用する。

**理由:** 型消去コストを回避し、Swift らしい API を実現する。

---

## ~~OQ-009: protocol メソッドの粒度~~ → **解決: D-9**

**決定:** 各レイヤーは単一 protocol を基本とし、実際に必要になった時点で分割する。

**理由:** 実用主義。Go SDK の ISP 5分割は過剰。最初からの過度な抽象化を避け、実用的な範囲内で設計する。ランタイム制御は AgentSession の責務とする（セッション中のみ有効なため）。
